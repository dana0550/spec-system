# Instruction Set for Codex: Product Spec System (Markdown-Only)

## 0) Operating principles
- **Single Source of Truth (SSOT):**
  - `FEATURES.md` is canonical for *what* exists (IDs, names, status, hierarchy).
  - Each `features/F-…*.md` file is canonical for *how it works* (goals, scope, flows, acceptance criteria).
  - `MASTER_SPEC.md` is canonical for *product goals, principles, and release scope*.
  - `PRODUCT_MAP.md` is the navigable tree that mirrors `FEATURES.md`.
- **Stable IDs, flexible names.** Feature *IDs* never change; names and slugs may change. Keep aliases.
- **Propagation before pretty.** When a change request arrives: update SSOT → propagate → re-render indices → then refactor wording.

---

## 1) Target folder layout (create if missing)

```
docs/
  README.md
  MASTER_SPEC.md
  FEATURES.md
  PRODUCT_MAP.md
  GLOSSARY.md
  DECISIONS/
    ADR_TEMPLATE.md
    ADR-0001-sample.md      # example
  templates/
    FEATURE_TEMPLATE.md
  features/
    F-001-your-feature.md   # top-level features live here
    F-001.01-your-sub.md    # sub-specs use dotted child IDs
```

**Naming rules**
- Files are lowercase, hyphen-separated slugs.
- IDs:
  - Top level: `F-001`, `F-002`, …
  - Children: `F-001.01`, `F-001.02`, …
  - Grandchildren: `F-001.02.03`, etc.
- Never renumber IDs once assigned.

---

## 2) Global conventions (LLM-friendly)

- **YAML frontmatter** at top of *every* Markdown file.
- **Relative links only** (no absolute URLs).
- **Backlinks section** in every feature to list parents/children and incoming references.
- **Machine markers** (exact strings) that Codex can search/replace:
  - `<!-- AUTOGEN:TOC -->` table of contents
  - `<!-- AUTOGEN:CHILDREN -->` direct children list
  - `<!-- AUTOGEN:REFERENCES -->` incoming refs discovered via ID scan
- `<!-- AUTOGEN:IMPACT_MAP -->` code/spec linkage
- `<!-- AUTOGEN:API_CONTRACT_SUMMARY -->`
- `<!-- AUTOGEN:BUDGET_CHECKLIST -->`
- `<!-- AUTOGEN:SECURITY_CHECKLIST -->`
  - `<!-- DO NOT EDIT BELOW (AUTOGENERATED) -->` guardrail for generated sections

## 4) Propagation rules (what to update when)

| Change request | SSOT first | Then update | Autogen steps Codex must run |
|---|---|---|---|
| Add a new top-level feature | `FEATURES.md` (new row) | Create `features/F-XXX-<slug>.md` from template; link in `MASTER_SPEC.md` summary | Rebuild `PRODUCT_MAP.md`; backfill `children` of parent (none) |
| Add a sub-spec | `FEATURES.md` (new row with Parent ID) | Create `features/F-XXX.YY-<slug>.md`; add to parent’s *Children* | Rebuild `PRODUCT_MAP.md`; refresh backlinks |
| Rename a feature | `FEATURES.md` (Name, add old name to Aliases) | Rename file and update `name` + `aliases` in frontmatter | Scan & replace links by `id:` not by text; rebuild map/backlinks |
| Change status | `FEATURES.md` | Update frontmatter `status` | Rebuild map; add note to master spec’s Release Scope if affected |
| Deprecate | `FEATURES.md` (`status: deprecated`) | Add banner in spec body; keep file (don’t delete) | Remove from active map section; keep in “Deprecated” section |
| Re-parent | `FEATURES.md` (Parent ID) | Update frontmatter `parent` | Rebuild map; update parent/child sections |
| Update code touching a feature | source files (tag with [F-XXX][R#?]) | feature spec (IMPACT_MAP), tests, contracts | Rebuild maps/backlinks; refresh IMPACT_MAP |

> Codex must **never** change IDs. If a feature’s concept is split, create new IDs and mark the original as `deprecated` with pointers.

---

## 5) Integrity checks Codex must run on every edit
1. **Index parity:** Every ID in `FEATURES.md` has exactly one spec file; every spec file’s `id` exists in `FEATURES.md`.
2. **Hierarchy validity:** All `parent` values exist; no cycles; children sets match dotted IDs.
3. **Link audit:** All relative links resolve; any renamed slug gets updated links.
4. **Backlinks refresh:** Recompute `<!-- AUTOGEN:REFERENCES -->` by scanning for `[F-…]` occurrences and IDs in frontmatter.
5. **Map rebuild:** Regenerate `PRODUCT_MAP.md` from `FEATURES.md`.
6. **Timestamps:** Touch `last_reviewed` fields and `last_synced/last_rendered`.
7. **Code parity:** Every R#/AC# has ≥1 code tag; every code tag references a valid F-ID.
8. **Contract parity:** All paths/fields referenced by feature text exist in contracts; breaking changes require ADR.
9. **Rollout readiness:** if status=active then flag exists, kill switch defined, guardrails present.

## 6) Codex workflows (copy/paste these prompts)

### 6.1 Bootstrap the structure
> **System prompt for Codex**  
> “You are a meticulous documents maintainer. Operate only in `docs/`. Respect SSOT rules: `FEATURES.md` → propagate to feature specs → regenerate `PRODUCT_MAP.md` and backlinks. Use the templates and frontmatter schemas exactly. Never change feature IDs. Always run integrity checks after edits and include a short `docs: …` commit message summarizing impacted IDs.”

> **User task: Bootstrap**  
> “Create the folder tree from Section 1, add `README.md` describing how to use this system (1 page), stub `MASTER_SPEC.md`, empty `FEATURES.md` table with headers, and `PRODUCT_MAP.md` placeholder. Add `templates/FEATURE_TEMPLATE.md`, `DECISIONS/ADR_TEMPLATE.md`, and an example `GLOSSARY.md`.”

### 6.2 Add a new top-level feature
> “Add a new feature **‘<Name>’** as **active**. Assign next ID. Update `FEATURES.md`. Create the spec from the template with a good initial Summary/Goals/Requirements. Link it from `MASTER_SPEC.md` current release if relevant. Rebuild product map and backlinks. Return a diff summary.”

### 6.3 Add a sub-spec
> “Under **<Parent ID>**, add sub-spec **‘<Name>’**. Assign next dotted child ID. Update index, create the sub-spec file, update parent’s *Children* section, rebuild product map/backlinks, and run integrity checks.”

### 6.4 Rename a feature safely
> “Rename **<ID>** to **‘<New Name>’**. Add the old name to `aliases`. Rename the file slug accordingly. Replace link texts only where the rendered title should change; keep `[F-…]` ID references intact. Rebuild maps/backlinks and run integrity checks.”

### 6.5 Deprecate a feature
> “Mark **<ID>** deprecated in `FEATURES.md` and its frontmatter. Add a banner at top of its spec with ‘Deprecated’ and the replacement IDs if any. Remove it from the active section of `PRODUCT_MAP.md` and add to a ‘Deprecated’ section at the bottom.”

### 6.6 Re-parent a feature
> “Change **<Child ID>** parent to **<New Parent ID>**. Update `FEATURES.md` and frontmatter. Update parent/child sections, rebuild product map/backlinks, and verify no cycles.”

## 7) `README.md` (what Codex should write)
```markdown
# Docs System Guide

This repository uses a Markdown-only spec system designed for LLM agents.

- **Start here:** `MASTER_SPEC.md` (vision, release scope)
- **Canonical features list:** `FEATURES.md`
- **Tree view:** `PRODUCT_MAP.md`
- **Specs:** `features/*.md`
- **Decisions:** `DECISIONS/`
- **Templates:** `templates/`

### How to propose changes
1) Edit `FEATURES.md` first (IDs/names/status/parent).  
2) Create or update the corresponding `features/F-…*.md`.  
3) Rebuild `PRODUCT_MAP.md` and backlinks (automated by Codex).  
4) Ensure integrity checks pass.

### Style & structure
- Keep prose concise; move detail into **Requirements** and **Acceptance Criteria** lists.
- Prefer references like `[F-001]` rather than name-only links.
- Add to `GLOSSARY.md` when you introduce new terms.
```

## 11) Versioning & distribution

- **Semantic versioning.** The `version` in this file’s frontmatter follows `MAJOR.MINOR.PATCH`.
  - Increment **MAJOR** for incompatible structure changes that require manual migration.
  - Increment **MINOR** for additive, backward-compatible guidance.
  - Increment **PATCH** for typo fixes or clarifications.
- **Upgrade policy.** When downstream repos hold an older copy, upgrade only after reviewing release notes below and running the sync script.
- **Release notes.** Keep the list below up to date so integrators understand changes before syncing.
- **Sync script.** Run `python scripts/sync_instruction_set.py <dest-repo>` (see script help for options). The script:
  - compares the local `version` to the destination file’s version (if present),
  - prompts on downgrades,
  - copies `DOCS_SYSTEM_INSTRUCTION_SET.md` into `<dest>/docs/DOCS_SYSTEM_INSTRUCTION_SET.md` by default, or a custom path via `--target`.
- **Audit trail.** Record upgrades in the destination repo’s changelog/ADR so teams know when the instruction set changed.
