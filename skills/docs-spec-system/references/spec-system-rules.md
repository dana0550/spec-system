# Spec System Rules

Skill release: `docs-spec-system` v1.0.0 (2026-02-06).

## Canonical Model

- `docs/FEATURES.md` is the SSOT for feature IDs, names, lifecycle, ownership, and parent relationships.
- `docs/features/F-...*.md` files define behavior details (goals, requirements, acceptance criteria, traceability).
- `docs/MASTER_SPEC.md` captures product vision, constraints, and release scope.
- `docs/PRODUCT_MAP.md` is a rendered hierarchy from `FEATURES.md`.
- IDs are immutable after assignment.

## Target Layout

```text
docs/
  README.md
  MASTER_SPEC.md
  FEATURES.md
  PRODUCT_MAP.md
  GLOSSARY.md
  DECISIONS/
    ADR_TEMPLATE.md
  templates/
    FEATURE_TEMPLATE.md
  features/
    F-001-<slug>.md
    F-001.01-<slug>.md
```

## Naming And ID Rules

- Use lowercase, hyphenated file slugs.
- Top-level IDs: `F-001`, `F-002`, ...
- Child IDs: `F-001.01`, `F-001.02`, ...
- Grandchild IDs: `F-001.01.01`, ...
- Do not renumber existing IDs.

## Propagation Matrix

| Change | Update first | Then update | Required regeneration |
|---|---|---|---|
| Add top-level feature | `FEATURES.md` | New spec file, `MASTER_SPEC.md` summary | `PRODUCT_MAP.md`, backlinks |
| Add sub-feature | `FEATURES.md` | New child spec, parent `Children` | `PRODUCT_MAP.md`, backlinks |
| Rename feature | `FEATURES.md` (`Name`, `Aliases`) | Rename spec file slug, frontmatter `name` + `aliases` | Map + backlinks + links by ID |
| Change lifecycle status | `FEATURES.md` | Feature frontmatter `status` | Map render + scope notes |
| Deprecate feature | `FEATURES.md` (`status: deprecated`) | Deprecated banner in spec | Move from active map to deprecated section |
| Re-parent feature | `FEATURES.md` (`Parent ID`) | Child frontmatter `parent` | Map + parent/child sections |
| Update R#/AC# | Feature spec | Checklist + traceability table + changelog | Backlinks + integrity checks |
| ADR create/update | ADR file | Link ADR in related feature specs | Backlinks + decision references |

## Integrity Checklist (Run Every Edit)

1. Every ID in `FEATURES.md` maps to exactly one spec file.
2. Every feature spec `id` exists in `FEATURES.md`.
3. Parent IDs resolve and hierarchy contains no cycles.
4. Relative links resolve after slug or name changes.
5. `PRODUCT_MAP.md` matches current `FEATURES.md` hierarchy.
6. `AUTOGEN:REFERENCES` sections are refreshed from ID scans.
7. `last_reviewed`, `last_synced`, and `last_rendered` timestamps are updated.
8. R#/AC# checklists and traceability table align with current spec text.
9. Deprecated features remain documented but are isolated from active map output.

## Done Checklist

- `FEATURES.md` updated first.
- Spec files updated with correct frontmatter and stable IDs.
- `PRODUCT_MAP.md` rebuilt.
- Backlinks refreshed.
- Integrity checks completed and reported.
- Diff summary includes impacted feature IDs.
